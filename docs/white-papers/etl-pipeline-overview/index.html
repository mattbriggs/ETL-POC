    ETL Pipeline for Converting Multiple Formats to DITA 1.3¶

    Introduction¶

    Converting a wide variety of document formats into DITA 1.3 XML can be achieved with an Extract-Transform-Load (ETL) pipeline that leverages Oxygen XML Editor (and its CLI scripting capabilities) alongside tools like Pandoc for broad format support. This pipeline will take inputs such as Markdown, Microsoft Word (Docx), Apple Pages, reStructuredText, LaTeX, EPUB, HTML, and many others, and produce structured DITA content (Concept, Task, and Reference topics) assembled into DITA maps. The solution is designed with robust software engineering principles in mind:

    • Isolated concerns: The conversion process is broken into clear stages (extract, transform, load). Each stage has a single responsibility with explicit inputs/outputs, and intermediate artifacts are treated as immutable (never modified in-place).
    • Contracts over tools: Stages communicate via well-defined formats and schemas (e.g. DocBook XML or XHTML as intermediates, and DITA XML as output) rather than through ad-hoc in-memory structures. This ensures any component (Pandoc, XSLT, etc.) can be swapped as long as it respects the input/output schema contracts.
    • Deterministic output: Given the same inputs and config, the pipeline will always produce the same DITA output. Each step’s results can be content-hashed to verify repeatability oai_citation:0‡medium.com. This determinism helps with caching and debugging (if a step is re-run with identical input, a hash match can skip unnecessary processing).
    • Configuration-first: All mappings, taxonomies, chunking rules, and output options are driven by external config files (YAML/JSON) under version control. No magic constants are hard-coded – things like “which styles map to Task topics” or “how to split a file into multiple topics” are defined in config, making the process transparent and adjustable without code changes.
    • Graceful degradation: The pipeline is resilient to conversion failures. A problematic file or section will not halt the entire run – instead it will be quarantined for review while the pipeline continues oai_citation:1‡medium.com. For example, if one document cannot be fully converted (due to an unsupported format feature or a parsing error), that error is logged and the pipeline moves on with the rest, ensuring partial extraction doesn’t block overall progress.
    • Testable components: Each stage of the pipeline is designed to be independently testable. We maintain small sample inputs (fixtures) and expected “golden” output files for unit testing of each stage. This means we can run a stage in isolation (e.g. conversion of a Markdown snippet to intermediate XML) and compare it to the expected result, giving confidence that changes in the pipeline don’t introduce regressions.

    Using Python on macOS as the orchestrator, we can integrate Oxygen’s powerful XML processing (via its CLI or Saxon XSLT engine) with Pandoc’s broad format conversion capabilities. The end result will be batch-automated conversion and classification of documents into DITA 1.3 XML, complete with DITA maps, and with support for custom DITA DTD specializations if needed.

    Pipeline Overview¶

    Extract-Transform-Load Stages:
    1. Extraction (Input Conversion): Parse and convert source files from their native formats into a normalized intermediate representation. This stage handles a wide array of input formats – from lightweight markups (Markdown, reStructuredText, AsciiDoc, Org-Mode, etc.) to binary docs (Word .docx, RTF, ODT), HTML/XML formats, wikis, slide decks, and more – using appropriate tools. The goal is to produce a consistent XML or JSON structure from all inputs, easing downstream processing.
    2. Transformation (Mapping to DITA): Take the intermediate representation and transform it into DITA XML topics. This involves classifying and splitting content into DITA topic types (Concept, Task, Reference) and ensuring each topic follows the DITA 1.3 structure. It may use XSLT (with Saxon via Oxygen) or Python logic to map elements to DITA equivalents. This stage also assembles metadata and applies any taxonomies or mappings defined in the config (for example, ensuring certain sections become <steps> in a Task topic, etc.).
    3. Load (Assembly and Output): Save the resulting DITA topics to the desired output location and assemble a DITA map (.ditamap) that organizes all topics. The DITA map can reflect an outline or hierarchy (derived from the source structure or config rules) and ensures all topics are referenced. In this stage, we also perform final validation (using Oxygen’s DITA validation or DITA-OT checks) to verify that the output is compliant with the DITA DTD (including any custom specializations). Successfully converted files become part of the output set, while any files that were quarantined or failed conversion are documented in a report for later manual attention.

    Each stage reads from and writes to well-defined files, and by chaining these stages we achieve an end-to-end conversion. The entire pipeline can be run as a Python script (or within an orchestration framework) to handle batches of documents automatically.

    Below, we detail each stage and the tools/plugins involved, aligning with the guiding principles.

    Stage 1: Extraction – Multi-Format Input Conversion¶

    Objective: Ingest source files of various formats and convert them into a uniform intermediate format (or a small set of formats) for easier processing. This isolates the complexity of dealing with dozens of formats into one stage.

    • Broad Format Support via Pandoc: We utilize Pandoc as a universal converter to handle the “widest number of file types.” Pandoc is renowned for supporting a broad array of input and output formats. In fact, Pandoc’s library can read/write markdown (CommonMark, GFM), reStructuredText, Asciidoc, Org-mode, Emacs Wiki, Textile, txt2tags, Djot; HTML4/5; EPUB (v2/v3); FictionBook FB2; GNU TexInfo; roff (man, ms); LaTeX; ConTeXt; DocBook (v4/v5); JATS; TEI; OpenDocument XML; OPML; BibTeX/BibLaTeX; CSL JSON/YAML; RIS; Word .docx; RTF; ODT; Jupyter .ipynb; InDesign ICML; various wiki markups (MediaWiki, DokuWiki, XWiki, Jira, etc.); slideshow formats (PowerPoint, Beamer, reveal.js, etc.); CSV/TSV; and even PDF (via external tools) oai_citation:2‡hackage.haskell.org oai_citation:3‡hackage.haskell.org. This means nearly all the formats listed in the question can be ingested by Pandoc. For example: a Markdown file or an Org-Mode note can be parsed, a Word .docx or RTF document can be read, a LaTeX or ConTeXt file can be parsed, etc. Using Pandoc’s command-line in a Python subprocess (or the pypandoc library), we convert each input file to a chosen intermediate format such as DocBook XML or HTML5. Both DocBook and XHTML are XML formats with schemas, serving as “contracts” for the next stage.

    • Using Oxygen’s Batch Converter for Certain Formats: For some formats, we can leverage Oxygen XML Editor’s Batch Converter add-on via its CLI scripts for direct conversion to DITA or XML. Oxygen’s scripting toolkit (requires a scripting license) provides a batchConverter script that can convert HTML->DITA, Markdown->DITA, Word->DITA, Excel->DITA, Confluence->DITA, DocBook->DITA, among others oai_citation:4‡oxygenxml.com oai_citation:5‡oxygenxml.com. For instance, if a .docx (Word) file or Markdown file needs conversion, we could call Oxygen’s batch converter (via a shell call from Python) to get DITA XML directly. However, in this pipeline we might prefer converting to a neutral format like DocBook first (which preserves semantics better) and doing the DITA mapping in the next stage. The Oxygen converter is still useful for quick conversion or for formats Pandoc might not handle (e.g. an Excel to DITA table conversion).

    • Handling Binary/Proprietary Formats: Some formats (e.g. Apple Pages) are not directly supported by Pandoc or Oxygen. For these, the pipeline should include a pre-processing step: for example, Apple Pages documents can be programmatically exported to .docx or .pdf (using AppleScript or the Pages GUI) before feeding into the pipeline. Similarly, if a format is unsupported, document it in the logs and skip or convert via an intermediate step (graceful degradation).

    • Explicit I/O and Immutable Artifacts: Each input file is processed independently, and the output of this stage is an intermediate file (or files) stored in an “extraction output” directory. For example, Guide123.docx might be converted to Guide123.xml in DocBook format. These intermediate files are treated as read-only once written (immutable); if the pipeline runs again on the same input without changes, it will produce the exact same intermediate (ensured by deterministic conversion settings). We can even generate a hash (e.g. MD5 or SHA256) of the content of each intermediate file and store it – both for caching (to skip re-conversion if the source hasn’t changed) and for traceability (each intermediate artifact can be traced and verified).

    • Logging and Quarantining Errors: If the conversion of a particular file fails or yields low-confidence output, this stage will log the error and mark that file for review. For example, if Pandoc encounters an unsupported construct in a LaTeX file, we capture Pandoc’s error message. Rather than aborting, the script continues with the next file (unless the failure is critical). Files that couldn’t be converted (or with partial conversion) are moved to a “quarantine” list. We output any content that was parsed before the failure, if possible, and annotate it so it doesn’t break the pipeline. This quarantine strategy ensures the pipeline keeps running with valid data separated from invalid, instead of failing entirely oai_citation:6‡medium.com.

    By the end of Stage 1, we have a collection of intermediate XML/JSON files representing the original documents in a consistent structure. We also have metadata (like the original file name, source format, timestamps, etc.) to carry forward. This stage’s isolation means that everything downstream works with a uniform contract (e.g. DocBook XML), simplifying the next transformations.

    Stage 2: Transformation – Classifying and Converting to DITA Topics¶

    Objective: Transform the intermediate representation into DITA 1.3-compliant XML topics, applying the rules for concept/task/reference typing, chunking content into topics, and enforcing structural consistency. This is the heart of the ETL process, where content is semantically structured into DITA.

    • XSLT Mapping (Contract-Driven): Given our intermediate is XML (e.g. DocBook 5 or XHTML), an XSLT stylesheet can be used to systematically map elements to DITA equivalents. For example, DocBook <section> or HTML <h1> might become a DITA <topic>. We will develop XSLT transformations (using Saxon, the XSLT engine integrated with Oxygen) to handle this mapping. The XSL defines the contract between intermediate schema and DITA schema – e.g., how a DocBook <para> becomes a <p> in DITA, how a DocBook <informaltable> becomes a DITA <table>, etc. This approach keeps us tool-agnostic; if a different converter produced the same DocBook, the XSLT would still work (adhering to “contracts over tools”). Oxygen’s Batch Converter actually includes a ready-made XSLT for DocBook to DITA conversion oai_citation:7‡oxygenxml.com, which we can reuse or adapt within our pipeline. Using Oxygen’s CLI, we could invoke a pre-configured transformation scenario for DocBook->DITA, or run Saxon via command-line in the Python script to apply our custom XSL.

    • Topic Chunking Rules: Often one source document needs to be split into multiple DITA topics (to avoid monster topics and to fit DITA’s topic-based authoring paradigm). We implement chunking rules defined in the config. For example, the config might specify that top-level sections (<h1> in HTML or <chapter> in DocBook) become separate DITA topic files, and nested sections (<h2>, <h3> etc.) become nested topics or sections within the parent topic. In Markdown-based DITA authoring, each level-1 heading starts a new topic oai_citation:8‡dita-ot.org, and we can mirror that logic: our XSLT can generate a new <topic> element each time it encounters a top-level section, closing the previous one. The output might be a single DITA file with nested <topic> elements (which DITA allows), or multiple .dita files – depending on how we prefer to output them. A mapping config can control this, for instance: chunk_level: 1 could mean “split at first-level headings into distinct files.” All decisions are externalized to config so that adjusting chunking doesn’t require code changes.

    • Classification into Concept/Task/Reference: A key requirement is to route content into Concept, Task, or Reference topic types appropriately. DITA 1.3 has these base types: Concept (for conceptual/background info), Task (for procedural step-by-step instructions), and Reference (for factual or reference data like specs) oai_citation:9‡heretto.com. We use both config rules and content cues to classify topics:

    • Config-driven classification: The simplest method is to let the user/config specify the intended topic type per document or section. For example, the YAML config could list file patterns or section titles that map to a certain type (e.g., any document with “How to” in its title is a Task). Another config approach is by location: e.g., all files in a Procedures/ folder could default to Task. These mappings are read in at runtime and applied during transformation.

    • Pattern recognition: In addition, the transformation stage can inspect content to infer type. For instance, if in the intermediate we detect an ordered list of steps (1., 2., 3.) or words like “Step 1:” or presence of imperative verbs, that section likely represents a procedure -> generate a <task> topic with proper <steps> inside. Conversely, if a section contains a table of parameters or a list of reference facts, it might be a Reference topic. If it’s mostly descriptive paragraphs with no obvious steps, it’s probably Concept. These heuristics can be refined over time or even augmented with a simple ML classifier, but starting with regex/keyword rules is often effective.

    • Applying the classification: Once we determine a topic’s type, the XSLT will generate the appropriate root element (<concept>, <task>, or <reference> instead of a generic <topic>). The DITA-OT Markdown mechanism, for example, uses a convention where adding a {.task} class to a Markdown H1 triggers a Task topic in output oai_citation:10‡dita-ot.org. In our XSLT, we effectively replicate that outcome: e.g., if a topic is marked as Task, we produce the <task> element and inside it a <taskbody> containing <steps>, etc., structured according to the DITA Task model. We ensure that required subelements (like a Task’s <title>, <steps>, and at least one <step> with <cmd>) are populated. Similarly, Reference topics might consist of property tables or definition lists formatted into DITA <dl> or tables. Concept topics would have a simpler <section> structure under a <concept> body. The Heretto DITA guide summarizes it well: Concept topics present background info, Task topics provide procedural how-to instructions, and Reference topics contain detailed factual information (often tabular or list form) oai_citation:11‡heretto.com.

    • Preserving and Mapping Content Elements: Besides overall structure, the transform stage maps finer details: paragraphs, lists, tables, images, code blocks, etc., into appropriate DITA elements (while obeying DITA content models). For instance:

    • Paragraphs become <p> within the DITA topic body or within <context> or other sections as appropriate. The first paragraph might become a <shortdesc> if config indicates that (DITA concept and task allow a short description element).
    • Section headings could become <section><title>...</title> blocks inside concept/reference topics, or in tasks they might map to <context>, <result>, etc., if the config or content suggests (e.g., a heading "Results" could map to the <result> section of a Task topic type).
    • Ordered lists that appear to be procedural steps: if classified as a Task, we wrap them in <steps><step>…</step></steps> and put the action in a <cmd> element inside each step (DITA’s way of marking what the user does) and any explanatory text in <info> oai_citation:12‡dita-ot.org. Unnumbered lists remain <ul>/<li> which becomes DITA <ul><li> (or possibly <sl> for step-less lists if inside a task but not actual steps).
    • Tables are transformed to <table> with proper <tgroup><tbody><row> etc., aligning with DITA table model (which is similar to HTML).
    • Images are converted to DITA <image> elements (with href pointing to the media file and possibly <alt> text captured). We ensure figure captions from sources become either <title> inside an <fig> wrapper or other suitable markup.

    • Cross-references: If the source had cross-reference links, we need to convert those to DITA <xref> elements. The intermediate likely expresses links as HTML anchors or DocBook xlinks; our XSLT will detect references to other docs and convert them. (In DITA, cross-topic links use <xref href="otherTopic.dita">). We may need to adjust the href extensions if topics are split. All such link rewriting can be handled with data gathered during chunking (e.g., we know a source document “Guide123” became “Guide123_concept.dita” and “Guide123_task.dita”, etc., so we can update references accordingly).

    • Deterministic Transformation: The XSLT (or Python transformation code) is written to produce consistent output given the same input and config. There is no reliance on external state or random values. If we run this stage tomorrow on the same intermediate file, the DITA topic XML will be identical (barring timestamp metadata if any). We can enforce this by avoiding any nondeterministic functions and by sorting any ambiguous sequences (for example, if iterating over JSON keys, ensure a stable order). The principle "same inputs, same outputs" is strictly followed oai_citation:13‡medium.com, which not only aids debugging but also allows caching: we might skip this transform if we detect the intermediate file’s hash is unchanged from a previous run and the output DITA file already exists with matching hash.

    • Graceful Degradation in Transformation: If certain content cannot be perfectly mapped to DITA, we handle it gracefully. For example, DITA has no direct equivalent for some exotic constructs – say a complex table nested in a list might be tricky. Our XSLT can output a <fnote> or <draft-comment> with the original content as fallback, so the information isn’t lost. Additionally, if an entire section fails (e.g., XSLT throws an error on an unexpected element), we catch that (Saxon allows try-catch via XSLT 3.0 or we do it by segmenting the XSLT). The problematic section can be omitted or left as raw content in a <pre> inside a DITA <note> marked as needing attention. This way, the final DITA file is still well-formed and valid, and the pipeline continues, with the issue noted for later fix.

    By the end of Stage 2, for each original input we have one or more DITA topic files (.dita XML) typed as concept, task, or reference. For example, a single Word file might have yielded three DITA topics: a concept intro, a task procedure, and a reference table – all derived according to rules. Each DITA topic will include the proper <title> (from the source’s title), optional <shortdesc>, and the content in the appropriate sections. These topics conform to either the base DITA DTD or a custom specialization DTD if specified (for instance, if using a specialized information type, the root element might be a custom <myConcept> which extends <concept> – Oxygen allows defining custom frameworks for such cases).

    Stage 3: Load – DITA Map Assembly and Output Deployment¶

    Objective: Aggregate all the generated DITA topic files into a coherent collection, create DITA maps to organize them, and output the final artifacts to the designated location (file system, repository, etc.).

    • DITA Map Generation: A DITA map (*.ditamap) is essentially a TOC or structure file that references all topics. We automatically build a map to include the converted topics. The map could be flat or hierarchical, depending on input structure or configuration. For instance, if an input Word document had chapters and subchapters that we split into separate DITA files, the map will reflect that nesting with nested <topicref> elements. We use the info gathered during Stage 2 (the relationships between topics, original ordering, etc.) to order the <topicref> entries. The map’s title can be taken from the source project name or a config setting. We ensure each topicref has the correct linking attributes (e.g., linking to the topic’s file, and the @type attribute could be set to "concept/task/reference" accordingly for clarity). If the pipeline processes multiple top-level documents, we might create multiple maps or one master map that includes them as chapters (configurable). The output folder will contain the DITA map and all the .dita topic files (likely in subfolders if organized).

    • Applying Output Flavor Configurations: “Output flavors” in config could control variations in output. For example, you might have a config for different DITA editions (maybe one that produces Light DITA vs full DITA, or DITA 1.2 vs 1.3, or includes certain metadata for one type of output). Because our pipeline is config-driven, we could easily support multiple output flavors. For instance, if the config says output_flavor: "Lightweight DITA" we might produce MDITA (Markdown DITA) or HDITA (XHTML-based DITA) instead of full XML DITA. Or if output_flavor: "DITA 2.0" in future, we adjust the transformation accordingly. In this case, the user specifically requested plain DITA 1.3 (the standard XML-based DITA v1.3), so our pipeline ensures compliance with that version. If custom DTDs are to be used (e.g., a specialized concept type), those DTDs (or schemas) should be referenced in the Oxygen framework and the output documents should declare the proper DOCTYPE or XML namespace. We can allow the config to specify a doctype public identifier or schema location for the output files if needed, thereby supporting extended DITA frameworks.

    • Output to Specified Location: The pipeline writes all final artifacts to a target directory (which could be, for example, a folder ready to be imported into a CCMS or a Git repository). Each run could go to a timestamped folder or a versioned folder if needed (since artifacts are immutable, one could keep past runs separate). The names of output files are deterministic (e.g., based on input file names and maybe section titles), or a mapping can be provided in config (some systems want specific file naming conventions). Because we hashed inputs, we could also include a content hash in file names for uniqueness if required, but generally keeping original names is fine.

    • Validation and Post-Processing: After generation, we perform a validation pass. Oxygen’s scripting includes a “DITA Validate and Check for Completeness” operation oai_citation:14‡oxygenxml.com that can be run on the output map. We can invoke this to ensure all topic references are resolved, all required DITA elements are present, and the content meets DITA rules. Any validation error (e.g., a missing @href or a topic missing a title) would be reported. Ideally, our transformation logic avoids such errors, but this is a good safety net and aids testability. If validation finds issues, we log them. If critical (config can define what is critical), we might mark the pipeline run as failed. Otherwise, minor issues could be tolerated with warnings (graceful degradation, as long as output is mostly usable).

    • Integration with Downstream Tools: At this point, the DITA content is ready. If needed, we could call the DITA Open Toolkit to publish output (like HTML or PDF), but that’s beyond the scope of conversion. However, an automated pipeline might trigger such publishing as a next step. We might also package the DITA content (e.g., as a ZIP or deliver it to a CCMS). Because each topic and map is stored as a file, source control or further processing can treat them just like any DITA project. The pipeline can also output a conversion report (perhaps in JSON) summarizing: how many topics created, any errors or warnings, time taken for each stage, etc., which is useful for auditing and continuous improvement.

    With Stage 3 complete, we have achieved the end goal: the source documents in dozens of formats have been transformed into a structured DITA 1.3 knowledge base, organized via maps. Writers or engineers can now use Oxygen XML Editor to open these DITA files for manual touch-ups if needed, or run DITA-OT to produce final deliverables. The pipeline’s output is deterministic and version-controlled, so re-running it on updated source docs will yield predictable updates in the DITA content.

    Configuration-Driven Processing¶

    A cornerstone of this solution is the use of external configuration files (YAML/JSON) to drive behavior:

    • Mappings and Taxonomies: The config defines how source content maps to DITA structures. For example, a style-to-element map for Word: (“Heading 1” style -> Topic Title, “Code Block” style -> <code> in DITA, etc.), or for Markdown: (a YAML metadata field ditatype: task could indicate the whole file is a Task). It can also contain a taxonomy of content types or product categories to assign metadata in DITA (for instance, tagging topics with audience or category metadata based on file location).
    • Chunking Rules: As mentioned, config controls how to split content. E.g. chunk_by: "h1" or a numeric level. It could even allow more complex rules like “split before any H2 that has style ‘TopicStart’” or “every slide in a PowerPoint becomes a separate topic”. These rules are read by Stage 2 to actually split nodes.
    • Output Options: The config can list output preferences such as the DITA doctype (concept vs task vs reference or a custom one). It may allow defaulting everything to concept unless specified, or using certain keywords to auto-detect. It can also hold the path for output, naming conventions, etc. Additionally, if multiple outputs are needed (say, one set of DITA topics in English and another in a different language or a lightweight format), config can specify multiple output flavors to run sequentially or in parallel.

    • Example Config Snippet (pseudo-YAML):

      source_formats:
  default_intermediate: "docbook5"
  treat_as_markdown: [".md", ".markdown", ".mdown"]
  treat_as_docx: [".pages"]  # e.g., we might rename .pages to .docx after conversion
classification_rules:
  by_filename:
    - pattern: "*_guide.*", topic_type: "concept"
    - pattern: "HOWTO_*", topic_type: "task"
  by_content:
    - pattern: "Step [0-9]+", topic_type: "task"
    - pattern: "Reference", topic_type: "reference"
chunking:
  level: 1  # split at first-level headings
  nested_topics: true  # allow nested <topic> in a single file for sub-sections
dita_output:
  dita_version: "1.3"
  use_specialization: false  # or name of specialized DTD
  output_folder: "out/dita"
  map_title: "Documentation Set"
      This illustrates how the pipeline might know what to do without hardcoding logic. The Python script reads these settings and the XSLT can also get parameters (Saxon allows passing parameters from command line, so we could pass chunkLevel=1 etc. to the stylesheet).

    • Version Control: The config files themselves are versioned (e.g. in Git). Any change to a mapping or rule is tracked. This ties into determinism and testing: if the output changed, we can pinpoint if it was due to a config change or a code change. Also, one can maintain different config versions for different projects or document sets.

    Using a config-first approach makes the system flexible – if tomorrow you decide that a certain type of content should be classified differently, you update config rather than code. It also decouples what we want to achieve from how the code does it, aligning with the principle of contracts (the config is essentially a contract for behavior).

    Automation and Orchestration¶

    A Python-based orchestration will coordinate these stages, ensuring the pipeline runs smoothly end-to-end:

    • Python Orchestration Script: A single Python script (or package) will implement the pipeline as a sequence of functions or classes corresponding to stages. For instance, extract_stage(input_path) -> intermediate_path, transform_stage(intermediate_path) -> [topic_files], load_stage(topic_files) -> ditamap_file. We keep these functions small and focused (isolate concerns). The main script will iterate over input files (supporting batch mode by reading a folder or a manifest file). Python’s rich standard library and third-party modules help here: e.g., use subprocess to call Pandoc or Oxygen CLI tools; use lxml or xml.etree if any inline XML tweaks needed in Python; use PyYAML or json to parse config files. We also use logging to track progress and issues.

    • Integration with Oxygen and Saxon: To run XSLT (for DocBook-to-DITA or similar), we have a few choices. One is to call Oxygen’s command-line Transform script, which can apply a predefined transformation scenario. Another is to call Saxon directly. Saxon HE (open-source) can be invoked by executing a .jar – e.g., java -jar saxon-he.jar -s:input.xml -xsl:transform.xsl -o:output.xml param1=.... We can embed such a call in Python. Since the user has Oxygen, they might also use the Oxygen SDK to integrate more tightly, but that typically involves Java programming. In Python, calling the external process is simpler. Oxygen’s Batch Converter script (as noted in Stage 1) is similarly called via shell (e.g., sh batchConverter.sh -i my.docx -if docx -o out -of dita). Our script can dynamically decide: “if format is docx and we trust Oxygen’s converter, call that directly to get DITA.” Or use Pandoc first, etc.

    • Parallel Processing: If dealing with a large number of files, we could parallelize the extraction stage easily (since each file is independent). Python’s concurrency (multiprocessing or multithreading for I/O bound tasks) can speed up conversion, especially when calling external tools. Care must be taken to not overwhelm CPU or memory (e.g., limit to a few parallel Pandoc instances). The transform stage could also potentially run in parallel for independent files, but if it’s XSLT via Saxon, it might be simpler to do sequentially unless using multiple Saxon instances. The orchestration could be managed by a job queue or just a loop with asynchronous calls. Regardless, parallelism should not compromise determinism – order of outputs doesn’t matter if each is complete, but for logging consistency and final map ordering, we might collect results and sort them by name.

    • Optional Workflow Frameworks: For more complex pipelines or integration into a larger system, one could use a workflow tool like Apache Airflow, Luigi, or Prefect to define the ETL steps and handle scheduling, retries, etc. For example, Luigi (a Python framework) could have tasks like ConvertToDocBookTask, TransformToDITATask, etc., with dependencies. This adds robustness (automatic rerun on failure, easier monitoring). However, if the pipeline is straightforward, a custom Python script or Makefile might suffice. Another lightweight option is using DVC (Data Version Control) which can manage pipeline stages with outputs tracked by hashes, aligning well with our approach of hashing each step’s outputs (DVC can skip stages if input/output unchanged). The user can choose the complexity based on need – since the question suggests a script, the answer focuses on that, but indeed these frameworks could be introduced as the pipeline scales.

    • Integration Testing the Pipeline: The orchestrator can include a self-test mode where it runs through known sample files. We maintain a directory of test inputs (covering each format type, e.g., a sample Markdown file, a sample Docx, a sample HTML) and expected DITA outputs (perhaps hand-validated). The script can iterate through these and diff the pipeline-generated output against the expected “golden” files. Any differences signal a regression. This automated test can be run whenever the pipeline code or config is updated. Since each stage is modular, we can also test stages in isolation: e.g., feed a sample Markdown to Stage 1 and verify the intermediate DocBook matches expectation (this tests Pandoc integration and our handling of it). Then feed that intermediate to Stage 2’s XSLT and verify the DITA result. Having these tests aligns with the “every stage has fixtures and golden files” principle, ensuring the system is reliable and changes are intentional.

    • Deterministic Behavior & Logging: The orchestrator enforces determinism by, for example, sorting the list of files to process (so the run order is always the same, avoiding random order issues). It can incorporate content hashing: after each stage, compute the hash of outputs and store them (perhaps in a manifest JSON). If a subsequent run finds an identical hash for inputs, it can skip reprocessing that stage to save time (this caching is optional but powerful for large docs). All actions are logged with timestamps and each file’s lifecycle is traced (e.g., “File X converted to DocBook, hashed = 123abc, then transformed to concept.dita and task.dita”). Logging also records any anomalies (like “Image in file Y was skipped due to conversion limitation”). This comprehensive logging, combined with consistent results, makes the pipeline auditable and easier to maintain.

    In summary, the automation ties everything together, ensuring that using Oxygen and Pandoc under the hood is seamless. The user runs the Python script (or it’s triggered by some CI job), and the pipeline outputs DITA content with no manual intervention. The modular design makes it easier to extend (e.g., adding a new input format later means mostly adding a new conversion rule in Stage 1 and maybe an XSLT template in Stage 2, without rewriting the entire system).

    Testing and Validation¶

    To guarantee quality, we incorporate multiple layers of testing and validation:

    • Unit Tests for Conversions: For each supported input format, create a minimal test file (for example, a Markdown snippet with a heading, a list, etc.; a small .docx with known content; an HTML sample). We then verify Stage 1 correctly produces the intermediate. This can be done by comparing the actual intermediate file to an expected file stored in a tests/expected/ directory. Because the conversion might include variable data (like timestamps or generated IDs), our comparison can ignore those or normalize them (e.g., remove id attributes before comparison if they are randomly generated). Ensuring Stage 1 is solid means Pandoc or Oxygen’s converters are working as we assume.

    • Unit/Integration Tests for Transformation: Similarly, test the XSLT transform with known intermediate inputs. We might craft a small DocBook XML input that covers various elements (paragraph, list, table, etc.), run our XSLT and compare the output to a golden DITA file. This tests our mapping logic. We include cases to ensure concept topics, task topics (with steps), and reference topics are generated as expected when certain cues are present. If we have a rule like “the word 'Steps' in a title triggers a Task,” we include that scenario. By having these tests, any change to the XSLT or config that alters output will be caught.

    • End-to-End Test: Take a representative sample document (maybe one in Markdown and one in Word), run the entire pipeline on it, and verify the final DITA map and topics match an expected set. This ensures the stages work in concert correctly.

    • Validation with DITA-OT/Oxygen: As mentioned, after generating DITA, we run a completeness check. Oxygen’s DITA validation will flag any structural errors. Our test suite can automatically run this and fail if validation errors are found. This way we never ship a pipeline that produces invalid DITA. We can also incorporate DITA-OT’s validator (the DITA-OT has a command for validating maps/topics) as a double-check.

    • Golden Files and Regression Control: The “golden files” approach means we treat expected outputs as part of the spec – if a change in code causes differences, we review if it’s an intended improvement (then update the expected files) or a bug. This provides high confidence in pipeline consistency, crucial for user trust (no unexpected doc changes).

    • Performance Testing: If needed, test the pipeline on a large batch to ensure it scales (and tune parallelism or memory as appropriate).

    By thoroughly testing at each stage, we adhere to the testable principle. The pipeline is not a black box – it’s a series of small boxes we can open and validate. This also makes onboarding new developers or writers to the system easier, since they can see concrete examples of input vs. output at each step.

The proposed process emphasizes maintainability and reliability: each stage has a single purpose and clear interfaces, configurations define behavior (making it easy to adjust mappings or add formats), and the system tolerates imperfect inputs without collapsing.
